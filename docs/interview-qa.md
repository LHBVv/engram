## 一、项目动机与定位类

### Q1: ChatGPT / Claude 都自带记忆了,你这个项目还有什么意义?
**A 要点**:
1. 区分两个市场:自带记忆服务的是"消费产品的用户";而开发者调用的 **API 是无状态的**,每个做 AI 应用的团队都要自己解决记忆问题——消费产品的功能 ≠ 开发者可用的基础设施(类比:微信有云端聊天记录,不解决"我开发 App 需要消息存储")。
2. 市场已验证:Mem0 / Zep / Letta 拿融资、被大量集成,证明独立记忆层是真需求。原因:数据主权(用户记忆是企业核心资产,不愿存在模型厂商)、跨模型可移植(换供应商记忆不丢)、可控可定制、多 agent 共享同一份记忆。
3. 定位站在平台吸收不掉的地方:企业级可控性(策略可配置、审计日志、自托管),而非"帮你存聊天记录"。
**深挖点**:如果被追问"OpenAI 出了 stateful API 怎么办"→ 承认简单记忆可能被平台吸收,但记忆层作为企业的"用户资产层"有独立存在的架构理由(资产要握在自己手里);且对我个人,项目价值在解决难问题的过程,不在商业存活。

### Q2: 为什么不直接把全部对话历史塞进 context?现在模型 context 都很长了。
**A 要点**:成本(token 随历史线性增长,每轮都重复付费)、延迟(长 prompt 首 token 慢)、质量(lost in the middle,长 context 中间信息利用率低)、上限(再长也有限,重度用户数月历史必然超)。选择性抽取 + 检索是把 O(全部历史) 压到 O(top-k 相关) 的方案。
**深挖点**:被问"long context 一直在变便宜变强怎么办"→ 承认趋势,但结构化记忆额外提供了 context 塞不了的能力:冲突消解后的"当前正确状态"、可审计、可管理(用户删除权)。

## 二、核心设计类

### Q3: 为什么读是同步的、写是异步的?
**A 要点**:读在对话延迟的关键路径上(用户在等回复),必须快(目标 P95<100ms);写发生在对话结束后,抽取+消解要调 LLM(秒级),用户感知不到。不对称的路径 → 不对称的设计:读侧优化检索性能,写侧优化抽取质量,独立扩展。附带:读路径 fail-open(服务挂了返回空,不阻塞接入方)。
**深挖点**:异步写的一致性问题——"刚说完的话立刻问,记忆还没写进去怎么办?"→ 本轮对话内容本来就在当前 context 里,记忆服务负责的是跨会话;写入延迟秒级,跨会话场景下不可感知。

### Q4: 冲突消解怎么做?LLM 判错了怎么办?
**A 要点**:候选事实 → 向量召回相似已有记忆 → LLM 判定 ADD/UPDATE/DELETE/NOOP + confidence。防御三层:(1) 保守策略,低置信度降级为 ADD(宁可冗余不可误删);(2) UPDATE/DELETE 都是归档不是物理删除,可恢复;(3) audit log 全量记录判定理由,可回溯可评测。
**深挖点**:阈值怎么定 → 用合成冲突评测集扫阈值,看 accuracy/误删率的 trade-off 曲线选(做完 Phase 3 后填入实际数字)。

### Q5: "什么都记"不行吗?为什么抽取阶段要过滤?
**A 要点**:记忆库的噪音直接毒害检索质量——垃圾记忆会占据 top-k 位置挤掉真正相关的。"不记什么"与"记什么"同等重要:排除临时状态、假设性讨论、模型自己说的话。这是 extraction 模块的核心难点之一。

## 三、技术选型类

### Q6: 为什么用 pgvector 而不是 Milvus / 专用向量库?
**A 要点**:见 ADR-001。核心:本项目每条查询都是"向量相似度 + 结构化过滤(user_id/tenant_id/status)+ 全文检索"的混合查询,单一 PostgreSQL 全覆盖,免去多存储一致性问题;项目规模(百万级向量以内)远未到 pgvector 性能瓶颈;运维半径小(三台机器 + 单人项目)。
**深挖点**:什么规模会迁移 → 向量数上亿、需要 GPU 索引或极致 QPS 时;但架构上存储层有抽象接口,迁移成本可控。
