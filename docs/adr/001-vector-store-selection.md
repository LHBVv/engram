# ADR-001: 向量存储选型 — PostgreSQL + pgvector

- 日期:2026-07-13
- 状态:已接受
- 决策人:Hongbin

## 背景

记忆服务的每条检索查询都是**混合查询**:向量相似度 + 结构化过滤(tenant_id / user_id / status=active)
+ 全文关键词 + 按时间排序。同时需要事务性写入(memories 与 audit_log 同事务)。
单人项目、三台机器(含 1 台 2C4G 级云服务器),运维预算极小。规模预期:十万级记忆条目以内。

## 候选方案

| 方案 | 优点 | 缺点 |
|---|---|---|
| **PostgreSQL + pgvector** | 单库覆盖向量+结构化+FTS+事务;混合过滤天然支持;运维半径最小;HNSW 索引在百万级以下性能足够 | 极大规模(上亿向量)性能不及专用库;无 GPU 索引 |
| Milvus / Qdrant(专用向量库) | 大规模性能强、向量功能丰富 | 需额外维护一个存储;结构化数据仍要 PG → 双写一致性问题;对本项目规模过度设计 |
| ChromaDB | 上手最快(testgen 时期已了解) | 偏嵌入式/原型定位,多租户过滤、事务、生产部署能力弱 |
| Elasticsearch(向量+FTS) | 混合检索一体 | 资源占用重(云服务器扛不住),事务能力弱 |

## 决定

选 **PostgreSQL 16 + pgvector**,FTS 用 PG 内置全文检索。

关键理由排序:
1. 混合查询是本项目检索的本质形态,单库最优
2. memories + audit_log 需要同事务写入,关系库是正解
3. 规模远未到 pgvector 瓶颈(HNSW 百万级以下延迟可控)
4. 学习价值:深入 PG(索引、FTS、事务)比学一个专用库 API 对后端求职更值钱

## 预期代价与迁移预案

- 接受:上亿向量规模下需要迁移(本项目达不到)
- 缓解:存储层做 Repository 抽象接口,检索引擎不直接依赖 SQL 细节,保留迁移可能
- 中文 FTS 的分词是已知坑(PG 默认不支持中文分词)→ 届时评估 zhparser / jieba 预分词方案,单独开 ADR

## 面试要点(蒸馏用)

"为什么不用专用向量库"→ 答案核心不是"pgvector 更好",而是**"我的查询形态是混合过滤,
数据有事务需求,规模在 PG 舒适区内——为不存在的规模引入多存储架构是负优化"**。
体现的是按需选型而非追新。
